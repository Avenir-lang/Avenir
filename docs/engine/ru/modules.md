# Модули и импорты

Этот документ описывает загрузку модулей и разрешение импортов в
`internal/modules`.

## Обзор

Модули определяются через декларацию `pckg` и импортируются по dotted path.
Загрузчик парсит entry-файл, рекурсивно разрешает импорты, объединяет
многофайловые модули и проверяет правила файлов/структур.

Ключевые файлы:

- `internal/modules/loader.go`
- `internal/ast/ast.go`

## Поток загрузки

`LoadWorld(entryFile)`:

1. Разрешает путь к entry-файлу.
2. Определяет `projectRoot` как директорию entry-файла.
3. Загружает entry-модуль и рекурсивно его импорты.
4. Обнаруживает циклы импортов.

Загруженные модули хранятся в `World`, ключ — полное имя модуля.

## Разрешение файлов

Для модуля `app.utils`:

1. Папочный вариант: `app/utils/utils.av`
2. Плоский файл: `app/utils.av`

Для модулей `std.*` логика такая же, но с корнем `std/`.
Если модуль не найден в корне проекта, загрузчик использует корень репозитория.

## Многофайловые модули

Если модуль импортирован как папка, загружаются все `.av` файлы директории и
объединяются в один AST (`mergePrograms`).

## Соответствие файл ↔ структура

Если файл содержит структуры, хотя бы одна структура должна совпадать с именем
файла (`Foo.av` должен содержать `struct Foo`).

Проверка выполняется в `validateFileStructMapping`.

## Алиасы импортов

По умолчанию локальное имя импорта — последний сегмент пути, но можно задать
алиас:

```avenir
import std.http.client as http;
```

Type checker добавляет в scope и алиас, и полное имя модуля, чтобы поддерживать
qualified-типы (`net.Socket`).

## Примечания

- Циклы импортов обнаруживаются и возвращаются как ошибки.
- Все файлы одного модуля должны иметь одинаковый `pckg`.
