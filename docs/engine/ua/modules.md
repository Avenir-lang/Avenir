# Модулі та імпорти

Цей документ описує завантаження модулів і резолв імпортів у
`internal/modules`.

## Огляд

Модулі визначаються декларацією `pckg` і імпортуються за dotted path.
Завантажувач парсить entry-файл, рекурсивно резолвить імпорти, об'єднує
багатофайлові модулі та перевіряє правила файлів/структур.

Ключові файли:

- `internal/modules/loader.go`
- `internal/ast/ast.go`

## Потік завантаження

`LoadWorld(entryFile)`:

1. Резолвить шлях до entry-файлу.
2. Визначає `projectRoot` як директорію entry-файлу.
3. Завантажує entry-модуль і рекурсивно його імпорти.
4. Виявляє цикли імпортів.

Завантажені модулі зберігаються у `World`, ключ — повне ім'я модуля.

## Резолв файлів

Для модуля `app.utils`:

1. Папковий варіант: `app/utils/utils.av`
2. Плоский файл: `app/utils.av`

Для модулів `std.*` логіка така сама, але з коренем `std/`.
Якщо модуль не знайдений у корені проєкту, завантажувач використовує корінь репозиторію.

## Багатофайлові модулі

Якщо модуль імпортовано як папку, завантажуються всі `.av` файли директорії
і об'єднуються в один AST (`mergePrograms`).

## Відповідність файл ↔ структура

Якщо файл містить структури, хоча б одна структура має збігатися з назвою файла
(`Foo.av` має містити `struct Foo`).

Перевірка виконується у `validateFileStructMapping`.

## Аліаси імпортів

За замовчуванням локальне ім'я імпорту — останній сегмент шляху, але можна задати
аліас:

```avenir
import std.http.client as http;
```

Type checker додає в scope і аліас, і повне ім'я модуля, щоб підтримувати
qualified-типи (`net.Socket`).

## Примітки

- Цикли імпортів виявляються і повертаються як помилки.
- Усі файли одного модуля мають мати однаковий `pckg`.
