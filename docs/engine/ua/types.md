# Типи і перевірка типів

Цей документ описує статичну систему типів і checker у `internal/types`.

## Огляд

Перевірка типів явна і статична: всі змінні та параметри повинні мати оголошені типи. Checker валідує структуру програми, резолвить імена та формує binding-метадані для IR-компілятора.

Ключові файли:

- `internal/types/types.go` (визначення типів)
- `internal/types/checker.go` (перевірка типів)

## Види типів

### Базові типи

`int`, `float`, `string`, `bool`, `bytes`, `void`, `any`, `error`.

### Списки

`list<T>` підтримує **кілька типів елементів**:

- `list<int>`
- `list<int, string>`

Всередині `List.ElementTypes` — список допустимих типів елементів. Літерали списків збирають унікальні типи елементів у порядку появи.

### Словники

`dict<K, V>` відображає ключі типу `K` у значення типу `V`. Скорочена форма `dict<V>` за замовчуванням встановлює `K` як `string`. Літерали словників виводять `V` як об'єднання типів значень за потреби.

Всередині `Dict.KeyType` дорівнює `nil` при використанні скороченої форми; хелпер `keyType()` повертає `String` у цьому випадку.

### Опціональні типи

`T?` представляє optional. `none` — єдине null-подібне значення, а `some(expr)` обгортає значення в optional.

Приклад:

```avenir
var n | int? = some(42);
var m | int? = none;
```

### Union-типи

`<A|B|C>` дозволяє значенню бути будь-яким з варіантів. Union-и нечутливі до порядку при порівнянні рівності.

Приклад:

```avenir
var mixed | <int|string> = 1;
```

### Структури та інтерфейси

- Структури — **номінальні**: імена визначають ідентичність.
- Інтерфейси — **структурні**: тип задовольняє інтерфейс, якщо надає всі необхідні методи з відповідними сигнатурами.

### Типи функцій

`fun(T1, T2) | R` для first-class функцій. Типи параметрів і повернення порівнюються структурно.

Функції можуть оголошувати типи помилок, що викидаються: `Func.Throws []Type`. Type checker валідує вирази `throw` проти оголошених throws і відстежує їх через поле `currentThrows`.

### Методи

Методи екземпляра зберігаються з їх `FunDecl` у полі `Method.Decl`. Це забезпечує:
- Обробку параметрів за замовчуванням для викликів методів
- Коректне відстеження параметра receiver у resolver-і
- Правильну IR-компіляцію викликів методів

Статичні методи обробляються як звичайні функції зі статичним receiver-ом.

### Future-типи (async)

Checker моделює async-повернення через `Future<T>`:

- `async fun ... | T` реєструється в scope як `fun(...) | Future<T>`.
- Усередині async-функції `return expr;` перевіряється проти `T` (внутрішній тип), а не `Future<T>`.
- `await expr` вимагає, щоб `expr` мав тип `Future<X>`, і повертає `X`.

Якщо `await` застосовано до не-future виразу, checker повертає помилку.

### Дженеріки

Checker підтримує явні дженеріки для структур і функцій:

- Узагальнені структури: `struct Box<T> { ... }`
- Узагальнені функції: `fun identity<T>(x | T) | T { ... }`
- Використання узагальнених типів: `Box<int>`
- Узагальнені виклики: `identity<int>(10)`

Аргументи типів явні. Виведення узагальнених аргументів типів наразі не реалізовано.

Вбудовані колекції (`list<...>`, `dict<...>`) — це параметричні вбудовані типи, які обробляються окремо від дженеріків користувача.

## Резолв імен

Checker заповнює scope:

- Builtin-функціями (тільки функції)
- Локальними оголошеннями
- Символами імпортованих модулів (за аліасом і повним шляхом)

Кваліфіковані типи (наприклад, `net.Socket`) резолвляться через додавання повного шляху модуля в scope під час обробки імпортів.

## Потік роботи checker-а

Високорівневий потік для мультимодульних world:

**Фаза 1a**: Створення scope і реєстрація builtins для кожного модуля. Попереднє оголошення імен структур (без полів).

**Фаза 1b**: Обробка імпортів, щоб крос-модульні типи були доступні при резолві параметрів функцій і полів структур.

**Фаза 1c**: Резолв полів структур, оголошення інтерфейсів, реєстрація функцій і методів з повними типами, реєстрація змінних верхнього рівня.

**Фаза 2**: Повна перевірка типів усіх операторів і виразів, включно з тілами функцій, з доступом до всіх імпортованих символів.

**Фаза 3**: Формування метаданих `Bindings` для IR-компіляції.

Для дженеріків оголошення та перевірка включають додаткові кроки:

1. Реєстрація узагальнених шаблонів як символів `GenericStruct` / `GenericFunc`.
2. При використанні `Name<TypeArgs>` резолв конкретних аргументів типів.
3. Інстанціювання конкретних типів/сигнатур через підстановку.
4. Реєстрація мономорфізованих структур/функцій під згенерованими іменами (наприклад, `Box$int`, `identity$int`).
5. Передача цих мономорфізованих записів через `Bindings`, щоб IR-компілятор включив лише інстанційовані версії.

## Правила присвоєння (основні)

- `any` приймає будь-який тип.
- Значення типу `T` присвоюється в `T?` (обгортається як `some` runtime-ом).
- `none` присвоюється будь-якому optional-типу.
- Union дозволяє присвоєння, якщо тип джерела відповідає будь-якому варіанту.
- Списки присвоюються, якщо кожен тип елемента сумісний з будь-яким цільовим типом елемента.
- Словники присвоюються, якщо і тип ключа, і тип значення сумісні з цільовими.
- Структури вимагають ідентичних імен (номінальна типізація).
- Інтерфейси вимагають повного збігу методів (структурна типізація).

Примітка: компілятор генерує `OpMakeSome` для явних літералів `some(...)` та для успішних шляхів опціонального ланцюжка (`?.`).

## Оператори

Checker забезпечує правила операторів, наприклад:

- `+` підтримує числове додавання та `string + string`.
- `-`, `*`, `/`, `%` вимагають числових операндів.
- Порівняння допускають числові операнди, `==`/`!=` допускають ширші порівняння.

## Builtins і методи

Builtins реєструються через `builtins.Meta` і додаються в scope як звичайні функції (`ReceiverType == TypeVoid`). Builtin-методи резолвляться при доступі до членів за типом receiver-а.

## Декоратори

Декоратори використовують синтаксис на основі виразів (`@<expr>`) і застосовуються під час ініціалізації.

Checker валідує декоратори на оголошеннях функцій через `checkDecorators`:

1. Перевірка типу виразу декоратора через `checkExpr(dec.Expr)`.
2. Якщо результат — `*Func`, перевірка що він приймає рівно один параметр-функцію, що відповідає типу декорованої функції, і повертає такий самий тип функції (`validateDecoratorFunc`).
3. Якщо результат — `*GenericFunc`, виведення аргументів типів із сигнатури декорованої функції через `inferDecoratorTypeArgs`, інстанціювання, перевірка та створення синтетичного `IdentExpr` з мономорфізованим іменем.

Оскільки декоратори — довільні вирази, параметризовані декоратори як `@cache(60)` обробляються природно: вираз `cache(60)` перевіряється як виклик, і тип його результату валідується як функція-декоратор.

Для методів екземпляра тип декорованої функції включає receiver як перший параметр (наприклад, `fun(Point, int) | int`), тому декоратор повинен приймати і повертати цей тип функції з receiver-ом.

Резолвлена інформація про декоратори зберігається в `Bindings.Decorators` як `[]*DecoratorInfo`, з ключем `*ast.FunDecl`. Кожен `DecoratorInfo` містить вираз декоратора (`ast.Expr`) та резолвлений тип функції.

IR-компілятор генерує синтетичну функцію `__init__`, яка застосовує декоратори під час ініціалізації модуля через `OpSetFunc`. Для кожної декорованої функції init-функція кладе оригінал як замикання, компілює вираз декоратора, викликає його та генерує `OpSetFunc` для заміни слоту функції. VM зберігає отримане замикання в `closureOverrides`, щоб `OpCall`, `OpClosure` та `OpSpawn` використовували декоровану версію. Це одноразова операція з нульовим оверхедом на кожен виклик.

## Варіативні дженеріки (TypePack)

Система типів підтримує варіативні параметри типів через `TypePack`:

- `TypePack` містить `[]Type` — пакет конкретних типів.
- `SubstituteType` розгортає `TypePack` у списках параметрів `Func`: якщо підставлений тип параметра є `TypePack`, його типи вставляються в список параметрів.
- `MonomorphKey` розгортає записи `TypePack` у рядок ключа.
- `typeOfTypeNode` обробляє `*ast.TypePackExpansion`, повертаючи `TypeVar`, який буде розгорнутий під час підстановки.

## Вихідні дані Bindings

`Bindings` включає резолв виразів/членів, узагальнену інстанціацію та дані декораторів:

- `Idents`, `Members`, `ExprTypes`
- `MonomorphizedStructs` (`monoName -> *types.Struct`)
- `MonomorphizedFuncs` (`monoName -> *ast.FunDecl`)
- `Decorators` (`*ast.FunDecl -> []*DecoratorInfo`)

IR-компілятор використовує мономорфізовані мапи для збору конкретних узагальнених інстанцій, а мапи декораторів — для генерації функції `__init__` для застосування декораторів під час ініціалізації.

## Помилки

Помилки типів збираються і повертаються як зріз `types.Error` з позиціями у вихідному коді.
